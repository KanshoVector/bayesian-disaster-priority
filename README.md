## 📂 プロジェクト構成（ディレクトリ構造）

各ファイルの役割を「UI層」「ビジネスロジック層」「データアクセス層」の3層構造で整理しています。

```tree
.
├── app.py           # [UI層] Streamlit画面構築・全体の統合司令塔
├── logic.py         # [ロジック層] 3つの推論モデルを統合するアンサンブル計算
├── bayes.py         # [ロジック層] ベイズ統計(共役事前分布) & MCMC(PyMC)サンプリング
├── bn_model.py      # [ロジック層] ベイジアンネットワーク(pgmpy)による推論
├── nb_model.py      # [ロジック層] ナイーブベイズによるテキスト分類補助
├── data.py          # [データ層] 架空の支援要請データを管理するモックDB
└── TECHNICAL.md     # [ドキュメント] 理論的背景と技術的負債の詳細仕様書
```

### 各レイヤーの役割

- **UI層 (`app.py`)**: ユーザー入力を受け取り、推論結果をグラフや数値として描画。
    
- **ビジネスロジック層 (`logic.py`, `bayes.py`, `bn_model.py`, `nb_model.py`)**: データの正規化から各数理手法の計算までを完結させる。疎結合に設計されており、特定のモデルの差し替えが容易。
    
- **データアクセス層 (`data.py`)**: データの永続化ロジックをカプセル化。将来的なSupabase/PostgreSQLへの移行時に、このファイルを改修するだけで対応可能。

## 🧠 設計メモ（Why一覧 ＆ 技術的負債の自己解剖）

### 1. なぜ3つのモデル（MCMC, BN, NB）のアンサンブルなのか？
災害時の「不確実性」に耐え、過去データゼロの状態からでも推論をスタートさせるため。
定性テキストを高速分類するナイーブベイズ（NB）、ドメイン知識による論理因果構造を担保するベイジアンネットワーク（BN）、歴史的な統計データの揺らぎを計算するMCMCサンプリングを"共通のインターフェース"である「確率（0.0〜1.0）」に正規化し、線形結合（加重平均）している。これにより、ブラックボックス化を完全に防ぎ、現場への「予測根拠の説明責任（Explainability）」を100%担保する設計とした。

### 2. なぜDBを導入せずインメモリ（data.py）なのか？
数理ロジックの有効性を検証するPoC（概念実証）のスピードを最優先し、環境依存を減らして一発デプロイするため。
ただし、データアクセス層（data.py）をインターフェースとして完全に分離・抽象化してあるため、モデル側のドメインロジックを1行も汚さずに、後からPostgreSQL（PostGIS等）/Supabaseへプラグイン式にリプレイスできる拡張性を担保している。

### 3. 現状のプロトタイプにおける「3つの致命的な数理的・技術的負債」
本リポジトリは短期的なPoCとしてLLMを駆動させ高速構築したため、プロダクション環境では即座に崩壊する以下の致命的な負債（Boundaries）を抱えており、現在の私の目線からは明確にバグおよび設計ミスとして認識している。

* **① 日本語前処理（形態素解析）の欠落によるテキスト分類（NB）の事実上の崩壊**
    * **現状の負債**: `CountVectorizer` を日本語テキストに対してそのまま適用したため、単語境界（わかち書き）が正しく認識されず、特徴空間が崩壊している。未知の語彙に対してはラプラススムージング（alpha=1.0）による一様確率を返すだけの死にコードと化している。
    * **本来あるべき設計**: 軽量な形態素解析器（`janome` や `MeCab`）をトークナイザーとしてインジェクションし、名詞・動詞・形容詞のみを抽出してベクトル化するか、LLMの埋め込みベクトル（Embedding）を用いたセマンティック解析へ移行すべきである。
* **② 事前分布ハイパーパラメータに対する線形補正の数理的厳密性の欠如**
    * **現状の負債**: `bayes.py` において、外部因子（被害規模等）を反映させるためにベータ分布の $\alpha, \beta$ に対して連続値を直接足し算（`past_severity * 2.0`）している。これは数理統計学的な一貫性を欠いた強引なヒューリスティクス（ハック）である。
    * **本来あるべき設計**: 外部因子自体を上位の確率変数としてモデルに組み込み、データからその影響度を同時にサンプリングさせる**「階層ベイズモデル（Hierarchical Bayesian Model）」**として構造化すべきである。
* **③ 連続値の強制3段階クリッピングによる境界線のエッジケース問題**
    * **現状の負債**: ベイジアンネットワークへの入力時、連続値を `0.33` や `0.66` で機械的に「高・中・低」へと離散化している。これにより、境界線付近のわずかなデータの揺らぎ（0.65と0.67など）で推論結果が不連続に跳ねる脆弱性がある。
    * **本来あるべき設計**: データの分布特性から K-means 等で動的にビン分割（Binning）を行うか、連続値をそのまま確率伝播できる**「ガウシアンベイジアンネットワーク」**を採用すべきである。

### 4. ユーザーが10倍になったらどこが壊れるか？（次期プロダクションアーキテクチャへの道標）
MCMC（PyMC）のハミルトニアンモンテカルロサンプリング（NUTS）がHTTPリクエストの主スレッド上で同期処理として走るため、同時リクエスト数が1を超えた時点でWebサーバー（Streamlit）が確実にフリーズ（ブロッキング）する。

* **解決策**: 
    本アプリの数理ロジックを実務に耐えうるシステムへ統合する場合、フロントエンドを **Next.js (Vercel)** に刷新し、推論ロジックを **FastAPI (Cloud Run)** へマイクロサービスとして切り離す。
    さらに、重いサンプリング計算をバックグラウンドに逃がすため、**Celery + Redis** による非同期分散タスクキューを導入し、リクエストに対して即座にタスクIDを返却、フロント側からポーリングまたはWebSocketで状態を同期する**イベント駆動型アーキテクチャ**へ移行する必要がある。
